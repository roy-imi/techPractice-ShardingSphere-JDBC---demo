# 实验与验收矩阵：固定季度槽位 + 部署期租户分库

这份文档把“我知道原理”变成可复现证据。默认部署季度为：

```properties
demo.sharding.current-quarter=2026Q3
demo.sharding.retained-quarter-count=3
```

因此本轮实验的在线窗口固定为 `[2026-01-01, 2026-10-01)`：

| 业务季度 | 状态 | 物理表 |
|---|---|---|
| 2026Q1 | ACTIVE，历史热数据 | `inspection_record_q1` |
| 2026Q2 | ACTIVE，历史热数据 | `inspection_record_q2` |
| 2026Q3 | ACTIVE，当前写季度 | `inspection_record_q3` |
| 2026Q4 | EMPTY（空闲待命），尚未激活 | `inspection_record_q4` |

每个实验都记录四项：输入、手算预期、Actual SQL、数据库侧 `server-id/schema`。只看到 HTTP 200 不能证明路由正确。

## 1. 验收证据链

```text
手算规则
  = /api/debug/routes/expected
  = ShardingSphere Logic SQL / Actual SQL
  = response.servedByServerId / servedByDatabase
  = 物理库表与主从复制状态
```

关键实现位置：

| 能力 | 实现 |
|---|---|
| 固定季度算法 | `com.example.assetinspection.algorithm.quarter.FixedQuarterShardingAlgorithm` |
| 租户分库算法 | `com.example.assetinspection.algorithm.tenant.TenantDatabaseShardingAlgorithm` |
| 标准版规则 | `src/main/resources/shardingsphere-standard.yaml` |
| 大客户版规则 | `src/main/resources/shardingsphere-tenant.yaml` |
| 部署期开关 | `ProductDataSourceConfiguration` + `sharding.enabled` |
| 热窗口配置 | `demo.sharding.current-quarter` |
| 业务边界保护 | `ShardingRangeValidator` |

- ds1 primary/replica 为 101/201；
- replica 的 IO/SQL 线程均为 `Yes`；
- replica 的 `read_only`、`super_read_only` 均开启；
- GTID 检查点已经追平；
- 一次性 `replication-setup` 显示 `Exited (0)` 是初始化成功。

### 2.3 在干净种子数据上验证物理槽位

```bash
./scripts/verify-physical-data.sh
```

通过标准：

- legacy 只有 `asset_legacy.inspection_record`；
- ds0、ds1 各有 q1～q4 四张固定季度表；
- q1/q2/q3 分别绑定 2026Q1/Q2/Q3 且为 ACTIVE；
- q4 为 EMPTY，且确实没有数据；
- 每条数据的自然季度与表后缀、YYYYQn 槽位绑定一致；
- 初始种子中 ds0 只有偶数租户、ds1 只有奇数租户；
- primary 与 replica 的表数据、槽位元数据一致。

注意：这个脚本的“偶/奇租户纯度”检查针对大客户拓扑的干净种子。后面若在标准版故意把奇数租户写入 ds0，同一数据卷再跑该检查会失败；这是预期现象，也正好说明部署形态不能直接热切换。

## 3. Legacy before 对照实验

### 3.1 `legacy` 在这里是什么意思

它是改造前的单库单表基线，也是迁移读取源，不是正式产品部署形态。启动：

```bash
mvn spring-boot:run -Dspring-boot.run.profiles=legacy
```

看到应用监听 `18080` 后，在另一个终端执行：

```bash
curl -sS \
  'http://localhost:18080/api/debug/routes/expected?tenantId=2&startDate=2026-01-01&endDateExclusive=2026-04-01'

curl -sS -H 'X-Tenant-Id: 2' \
  'http://localhost:18080/api/inspection-records?startDate=2026-01-01&endDateExclusive=2026-04-01&pageSize=20'
```

通过标准：

- `deploymentMode=LEGACY_SINGLE_TABLE`；
- `logicalDataSource=legacy_mysql`；
- `physicalTables=["inspection_record"]`；
- 响应中的 `servedByServerId=10`、`servedByDatabase=asset_legacy`；
- 控制台没有 ShardingSphere Actual SQL；
- Controller、Service、Mapper 不知道任何 q1～q4 物理表名。

记录这一组为 before 证据，然后用 `Ctrl+C` 停止应用。

## 4. 标准版实验：`sharding.enabled=false`

### 4.1 启动并确认选中了 standard YAML

```bash
CURRENT_QUARTER=2026Q3 \
SHARDING_ENABLED=false \
mvn spring-boot:run -Dspring-boot.run.profiles=product
```

启动日志应明确包含：

```text
产品部署模式已锁定为 SHARED_DATABASE
tenant database sharding enabled=false
shardingsphere-standard.yaml
```

`false` 只表示不按 tenant_id 分库，不代表关闭 ShardingSphere。季度分表、Snowflake 和读写分离仍然生效。

### 4.2 验证任意租户都路由 ds0

分别推演偶数和奇数租户：

```bash
curl -sS \
  'http://localhost:18080/api/debug/routes/expected?tenantId=2&startDate=2026-07-01&endDateExclusive=2026-08-01'

curl -sS \
  'http://localhost:18080/api/debug/routes/expected?tenantId=3&startDate=2026-07-01&endDateExclusive=2026-08-01'
```

两次都应返回：

```json
{
  "deploymentMode": "SHARED_DATABASE",
  "tenantDatabaseShardingEnabled": false,
  "logicalDataSource": "ds_0",
  "physicalTables": ["inspection_record_q3"],
  "writeTarget": "ds0_primary(server-id=100)",
  "normalReadTarget": "ds0_replica(server-id=200)"
}
```

这一步证明 `tenant_id` 在标准版仍是 SQL 隔离条件，但不参与数据库选择。

### 4.3 创建当前 Q3 数据

在线 create 必须使用当前季度 Q3。下面故意用奇数 tenant 3，证明标准版仍写 ds0：

```bash
curl -sS -X POST 'http://localhost:18080/api/inspection-records' \
  -H 'Content-Type: application/json' \
  -H 'X-Tenant-Id: 3' \
  -d '{
    "requestId":"standard-t3-2026q3-001",
    "assetId":30001,
    "assetCode":"STANDARD-PUMP-03",
    "inspectionPointId":6301,
    "inspectionPointName":"轴承温度",
    "recordDate":"2026-07-18",
    "inspectedAt":"2026-07-18T09:00:00",
    "status":"NORMAL",
    "measuredValue":62.5,
    "unit":"℃",
    "resultDescription":"标准版路由验收",
    "inspectorId":9301
  }'
```

首次应返回 201。复制响应中的 `id`，下文记为 `STANDARD_T3_ID`。重复执行前请修改 requestId，否则会得到 409 `DUPLICATE_REQUEST`。

Actual SQL 应包含：

```text
ds0_primary ::: INSERT INTO inspection_record_q3 ...
```

并且 create 事务内的回读仍在 ds0 primary，响应 `servedByServerId=100`。

### 4.4 普通读、事务读与 Hint 读

把 URL 中的 `STANDARD_T3_ID` 替换为真实 ID：

```bash
curl -sS -H 'X-Tenant-Id: 3' \
  'http://localhost:18080/api/inspection-records/STANDARD_T3_ID?recordDate=2026-07-18'

curl -sS -H 'X-Tenant-Id: 3' \
  'http://localhost:18080/api/inspection-records/STANDARD_T3_ID/strong?recordDate=2026-07-18'

curl -sS -H 'X-Tenant-Id: 3' \
  'http://localhost:18080/api/inspection-records/STANDARD_T3_ID/hint-primary?recordDate=2026-07-18'
```

| 请求 | 表 | server-id | 数据库 |
|---|---|---:|---|
| 普通读 | q3 | 200 | `asset_ds_0` |
| strong 事务读 | q3 | 100 | `asset_ds_0` |
| hint-primary | q3 | 100 | `asset_ds_0` |

如果普通读刚开始返回 404，而 strong 能查到，先等待复制追平；这是主从延迟，不是路由到了 ds1。

### 4.5 标准版跨 Q2/Q3

```bash
curl -sS -H 'X-Tenant-Id: 3' \
  'http://localhost:18080/api/inspection-records?startDate=2026-06-29&endDateExclusive=2026-07-19&pageSize=20'
```

通过标准：

- 只出现 ds0 replica；
- 只出现 `inspection_record_q2` 与 `inspection_record_q3` 两张表；
- 不出现 q1、q4 或任何 ds1 节点；
- 即使 q2 没有 tenant 3 结果，路由阶段仍会因范围覆盖 Q2 而执行对应物理 SQL。

### 4.6 证明不能直接把 false 热切为 true

停止应用后直接以大客户模式启动，但暂时不要迁移刚才的标准版 tenant 3 数据：

```bash
CURRENT_QUARTER=2026Q3 \
SHARDING_ENABLED=true \
mvn spring-boot:run -Dspring-boot.run.profiles=product
```

再用 tenant 3 查询 `STANDARD_T3_ID`。规则会改为 ds1，但数据仍在 ds0，因此返回 404。这个实验的结论不是“开关坏了”，而是：**拓扑变化必须伴随存量重分布、增量同步、对账和切流**。

为让后续大客户实验回到干净数据，先停止应用，然后执行本地重置：

```bash
./scripts/stop-databases.sh
docker compose down --volumes --remove-orphans
./scripts/start-databases.sh
```

`--volumes` 会永久删除本 Demo 的本地数据库卷。只应在确认不需要保留实验数据后执行；如果不想清空，可以换一套独立数据卷完成后续实验。

## 6. 季度热窗口保护实验

以下实验在 standard 和 tenant 模式都成立；区别只在数据库选择。

### 6.1 过期季度查询被拒绝

当前 Q3 的最早在线日期是 2026-01-01。请求 2025Q4：

```bash
curl -i -sS -H 'X-Tenant-Id: 3' \
  'http://localhost:18080/api/inspection-records?startDate=2025-10-01&endDateExclusive=2026-01-01&pageSize=20'
```

通过标准：HTTP 400、错误码 `DATE_RANGE_EXPIRED`，且没有 Actual SQL。系统不能把 2025Q4 悄悄路由到当前空闲的 q4。

### 6.2 未来季度查询被拒绝

```bash
curl -i -sS -H 'X-Tenant-Id: 3' \
  'http://localhost:18080/api/inspection-records?startDate=2026-10-01&endDateExclusive=2026-10-02&pageSize=20'
```

通过标准：HTTP 400、错误码 `DATE_RANGE_FUTURE_QUARTER`，且没有 Actual SQL。Q4 在 ACTIVATE 之前只是待命槽位。

### 6.3 普通 create 写 Q2 被拒绝

Q2 仍可查询和受控迁移，但实时新增只能写当前 Q3：

```bash
curl -i -sS -X POST 'http://localhost:18080/api/inspection-records' \
  -H 'Content-Type: application/json' \
  -H 'X-Tenant-Id: 2' \
  -d '{
    "requestId":"reject-create-q2-001",
    "assetId":20009,
    "assetCode":"Q2-REJECT",
    "inspectionPointId":6299,
    "inspectionPointName":"历史补录保护",
    "recordDate":"2026-05-18",
    "inspectedAt":"2026-05-18T09:00:00",
    "status":"NORMAL",
    "inspectorId":9299
  }'
```

通过标准：HTTP 400、错误码 `WRITE_NOT_CURRENT_QUARTER`，没有 q2 INSERT。若业务需要历史补录，应走单独的受控补录/迁移流程。

### 6.4 Q3 create 正常通过

把上面 requestId、recordDate、inspectedAt 改为全新值和 `2026-09-18`，应返回 201 并精确写入 q3。这个正反对照证明校验的是“业务季度状态”，不是简单禁写某张表。

### 6.5 recordDate 与 inspectedAt 不一致

```bash

  -H 'Content-Type: application/json' \
  -H 'X-Tenant-Id: 2' \
  -d '{
    "requestId":"date-mismatch-q3-001",
    "assetId":20010,
    "assetCode":"DATE-MISMATCH",
    "inspectionPointId":6210,
    "inspectionPointName":"日期一致性",
    "recordDate":"2026-09-30",
    "inspectedAt":"2026-10-01T00:01:00",
    "status":"NORMAL",
    "inspectorId":9210
  }'
```

通过标准：HTTP 400、`RECORD_DATE_MISMATCH`，没有任何物理写入。

## 9. 乐观锁、幂等和游标实验

### 9.1 幂等 requestId

连续发送两次 tenant、Q3、requestId 完全相同的创建请求。预期第一次 201，第二次 409 `DUPLICATE_REQUEST`。

局部唯一索引只保证同租户、同一季度槽位内唯一。若业务要求跨季度全局幂等，需要独立幂等表或缓存并设计一致性。

### 9.2 乐观锁

先 strong 查询 `TENANT_T2_ID` 并记住实际 `version`，然后用同一版本连续更新两次：

```bash
curl -i -sS -X PATCH \
  'http://localhost:18080/api/inspection-records/TENANT_T2_ID/result?recordDate=2026-08-08' \
  -H 'Content-Type: application/json' \
  -H 'X-Tenant-Id: 2' \
  -d '{"status":"REPAIRED","resultDescription":"验收修复","version":0}'
```

若当前版本不是 0，请替换为实际值。第一次应在 ds0 primary 的 q3 更新并把 version 加 1；第二次相同 version 应返回 409 `OPTIMISTIC_LOCK_CONFLICT`。

### 9.3 复合游标分页

为同一 tenant 在 Q3 创建至少 5 条不同日期/ID 的数据，然后：

```bash
curl -sS -H 'X-Tenant-Id: 2' \
  'http://localhost:18080/api/inspection-records?startDate=2026-07-01&endDateExclusive=2026-10-01&pageSize=2'
```

把返回的 `nextCursorDate`、`nextCursorId` 带入第二页：

```bash
curl -sS -H 'X-Tenant-Id: 2' \
  'http://localhost:18080/api/inspection-records?startDate=2026-07-01&endDateExclusive=2026-10-01&pageSize=2&cursorDate=NEXT_DATE&cursorId=NEXT_ID'
```

通过标准：两页无重复，顺序是 `record_date DESC, id DESC`；只传一个游标字段返回 400 `INCOMPLETE_CURSOR`。

## 10. 迁移与热窗口对账实验

迁移只在产品模式临时开启。这里选择大客户版，以验证偶/奇租户分别进入 ds0/ds1：

```bash
CURRENT_QUARTER=2026Q3 \
SHARDING_ENABLED=true \
mvn spring-boot:run \
  -Dspring-boot.run.profiles=product \
  -Dspring-boot.run.arguments=--demo.migration.enabled=true
```

如果应用已经运行，请先 `Ctrl+C` 停止，不能让两个进程同时占用 18080。

### 10.1 小批次游标回填

```bash
curl -sS -X POST \
  'http://localhost:18080/api/admin/migrations/legacy/batches?afterId=0&batchSize=2'
```

把响应的 `nextAfterId` 作为下一次 afterId，直到 `hasMore=false`。干净种子下通常依次推进到旧表最后一个 ID；不要猜游标，只使用上一次成功响应。

通过标准：

- 每批 `readRows <= 2`，游标严格递增；
- 旧数据位于 2026Q1/Q2，属于当前 Q3 配置下的 ACTIVE 热窗口；
- tenant 2 数据进入 ds0，tenant 3 数据进入 ds1；
- Q1 数据写 q1，Q2 数据写 q2；
- 旧 ID 原样保留，不重新生成 Snowflake ID；
- 重跑同一批时通过 `INSERT IGNORE` 计入 ignoredRows，不产生重复记录。

这里刻意说明：迁移允许 Q1/Q2，是因为它是受控历史回填；普通 create 仍只允许 Q3。

### 10.2 严格窗口对账

选择 tenant 2 的 2026 年 2 月；该窗口没有目标侧独立种子，适合严格 COUNT 对账：

```bash
curl -sS \
  'http://localhost:18080/api/admin/migrations/legacy/reconciliation?tenantId=2&startDate=2026-02-01&endDateExclusive=2026-03-01'
```

通过标准：

```text
legacyCount = 1
shardedCount = 1
shardedMinusLegacy = 0
matched = true
```

如果你在该窗口额外插入过数据，COUNT 不会严格相等；生产应使用迁移批次/来源标记，并进一步做状态聚合、数值汇总、分块校验和与抽样字段比对。

### 10.3 对账接口也受热窗口保护

以下过期窗口应返回 400，而不是把 2025Q4 错映射到 q4：

```bash
curl -i -sS \
  'http://localhost:18080/api/admin/migrations/legacy/reconciliation?tenantId=2&startDate=2025-10-01&endDateExclusive=2026-01-01'
```

未来窗口同样被拒绝：

```bash
curl -i -sS \
  'http://localhost:18080/api/admin/migrations/legacy/reconciliation?tenantId=2&startDate=2026-10-01&endDateExclusive=2027-01-01'
```

历史已归档数据应走单独的冷数据迁移/查询链路，不能借在线 Mapper 绕过槽位生命周期。

## 11. 槽位元数据与季度切换验收

本实验默认不执行清空，只观察状态：

```bash
# 当前种子状态是 2026Q3；先验证发布门禁在稳定态也能通过。
./scripts/quarter-rollover.sh release-check 2026Q3

docker compose exec -T ds0-primary mysql \
  -uroot -proot-password -D asset_ds_0 \
  -e 'SELECT slot_no, physical_table, bound_quarter, slot_status, expired_at, cleanup_after FROM inspection_quarter_slot ORDER BY slot_no;'
```

预期 q1/q2/q3 ACTIVE，q4 EMPTY。切到 2026Q4 的生产流程应是：

```text
PREPARE q4 为空且结构一致
  -> ACTIVATE：将 q4 元数据绑定为 2026Q4 / ACTIVE
  -> RELEASE-CHECK：绑定、数据归属、表结构和主从 GTID 通过
  -> 发布 demo.sharding.current-quarter=2026Q4 并滚动重启
  -> 验证 Q4 新读写路由
  -> EXPIRE 2026Q1 / q1
  -> 至少宽限 3 天
  -> 归档、COUNT/聚合/校验和、流量栅栏、可恢复备份、复制健康
  -> 人工批准 PURGE q1
  -> q1 进入 EMPTY（空闲待命），等待 2027Q1
```

3 天宽限期内四张表都可能有数据，但应用的正常在线路由仍只有 Q2/Q3/Q4 三个热季度；槽位元数据在 ACTIVATE 到 EXPIRE 之间会短暂有四个 ACTIVE。不要为了演示直接手工 TRUNCATE；清空是不可逆运维动作。

## 13. 面试前十秒手算

默认 current quarter 为 2026Q3。随机题：

```text
部署模式：tenant
tenant=17
range=[2026-06-29, 2026-07-02)
SQL=普通 SELECT
```

十秒内回答：

```text
17 mod 2 = 1 -> ds_1
范围跨 2026Q2/Q3 -> inspection_record_q2 + inspection_record_q3
普通非事务读 -> ds1_replica / server-id 201
两条物理 SQL 后需要归并
```

把模式换成 standard 时，tenant 17 也固定 ds_0，普通读为 server-id 200；把 SQL 换成 INSERT 时只能写 Q3 且走 primary；把日期换成 2025Q4、2026Q4 或把 create 换成 Q2，要能立即说出对应拒绝码。

## 14. 清理环境

先停止应用，再停止数据库容器但保留卷：

```bash
./scripts/stop-databases.sh
```

只有明确要清空全部本地实验数据并重新初始化时才执行：

```bash
docker compose down --volumes --remove-orphans
```

`--volumes` 不可恢复。季度过期表的生产清理更不能用这条命令替代正式的归档、对账和审批流程。
