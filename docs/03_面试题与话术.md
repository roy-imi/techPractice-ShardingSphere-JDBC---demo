# 面试题与话术：固定季度槽位 + 部署期租户分库

这份文档的目标不是让你背中间件名词，而是让你能把“为什么设计、如何路由、怎样证明、哪里有边界”讲成一条完整证据链。先完成 `docs/04_实验与验收矩阵.md`，再练下面的话术。

## 1. 先解释清楚 legacy

本项目里的 `legacy` 只表示 **改造前遗留架构的对照基线**：

- 一个 MySQL schema：`asset_legacy`；
- 一张物理表：`inspection_record`；
- 不经过 ShardingSphere-JDBC；
- 不做季度分表、租户分库或主从读写路由；
- 同时也是演示历史数据回填时的只读源库。

它不是“老版本仍要长期交付”，也不是第三种正式产品方案。正式交付只有两个部署形态：

1. 标准版：`sharding.enabled=false`，所有租户共享 `ds_0`，仍使用四张季度表和读写分离。
2. 大客户版：`sharding.enabled=true`，按固定键 `tenant_id` 分到 `ds_0/ds_1`，每库仍使用四张季度表和读写分离。

`legacy` 的价值是提供 before 对照和迁移来源，让你能证明业务代码如何从单表演进到逻辑表路由。

## 2. 先把简历原句讲准确

“统一管理 MySQL 主从集群”容易被理解为 ShardingSphere-JDBC 会创建复制、自动选主和完成 HA。面试时主动收窄：

> 这里的统一管理是指应用 JDBC 层统一配置多个 MySQL 主从数据源，并根据分片键、SQL 类型、事务状态或 Hint 完成库、表和读写节点路由。MySQL 复制关系、故障检测、主从切换、备份恢复仍由数据库基础设施负责。

“长期保留近 3 个季度共 9 个月”也需要说严谨：

> 在线窗口是当前季度加前两个完整季度。当前季度刚开始时覆盖略多于 6 个月，季度结束前接近 9 个月，所以更准确的说法是“完整覆盖最近半年，动态约 6～9 个月”。

“配置一键开启租户分库”需要补充边界：

> 这是部署启动参数，不是运行期热开关。客户首次部署前选择拓扑；有存量数据后若改变开关，必须按正式再分片迁移处理，不能只改配置重启。

## 3. 30 秒版本

### 参与过真实生产项目时

> 点巡检记录原来集中在单库单表。结合数据增长趋势和近半年高频查询特征，我们在 JDBC 层接入 ShardingSphere-JDBC，把时间维度设计为四张固定季度槽位表，在线只允许当前季度和前两个季度，第四张表常态为空闲待命；季度切换先激活新季度，再把最老季度标记过期，宽限 3 天并完成归档对账后才人工清理。产品交付上，标准版关闭租户分库，所有租户共享 ds_0；大客户版在部署时开启，按 tenant_id 分库。两种形态复用同一套业务代码，并对每个逻辑库组配置主从读写路由。效果只用真实压测和生产指标说明。

最后一句必须换成你自己的真实 QPS、P99、单表行数与资源数据；没有证据就不报数字。

### 仅用本 Demo 学习时

> 我用资产点巡检 Demo 复现了从 legacy 单库单表到 ShardingSphere-JDBC 的演进。表维度由 `FixedQuarterShardingAlgorithm` 把日期路由到 q1～q4，并用 `demo.sharding.current-quarter` 防止跨年复用读错；标准版选择 `shardingsphere-standard.yaml`，所有租户都进 ds_0，大客户版选择 `shardingsphere-tenant.yaml`，由 `TenantDatabaseShardingAlgorithm` 按 tenant_id 对 2 取模。通过 Actual SQL、数据库 server-id、过期/未来保护、主从延迟和迁移对账实验验证机制。它是学习复现，不把本地结果包装成生产业绩。

## 4. 两分钟版本

> 背景是资产管理系统点巡检模块。改造前所有租户、所有历史记录都在 `asset_legacy.inspection_record`。数据持续增长后，索引和热点集中在一个实例，在线查询、写入和归档相互竞争。业务访问又有两个稳定特征：每次请求都有租户上下文，高频查询主要关注最近半年。
>
> 表维度没有按无限增长的动态表名设计，而是固定四个日历季度槽位：Q1 永远对应 `inspection_record_q1`，依次到 q4。部署配置 `demo.sharding.current-quarter=2026Q3` 时，ACTIVE 窗口是 2026Q1、Q2、Q3，q4 为空闲。`FixedQuarterShardingAlgorithm` 不只计算季度后缀，还校验年份季度必须处于热窗口，因此 2025Q1 不会因为同样映射 q1 而误读 2026Q1 数据。在线新增只允许当前季度；前两个季度可查、可受控迁移，但普通补录不能绕过生命周期。
>
> 租户分库做成部署期产品能力。`sharding.enabled=false` 时，`ProductDataSourceConfiguration` 选择 `shardingsphere-standard.yaml`，所有租户共享 ds_0，但季度分表和读写分离仍然存在；`true` 时选择 `shardingsphere-tenant.yaml`，由 `TenantDatabaseShardingAlgorithm` 将偶数租户路由 ds_0、奇数租户路由 ds_1。开关只在启动时决定拓扑，不能在线刷新。分片键固定是 tenant_id，配置项只是防误配校验，不是任意换列名的能力。
>
> 每个逻辑库组是一主一从：写、事务内读和 Hint 强制读走 primary，普通非事务读走 replica。业务 SQL 始终访问逻辑表 `inspection_record`，Service 层要求 SQL 带 tenant_id 和 record_date 或时间范围。缺日期的教学反例会在目标库广播 q1～q4 四张表。
>
> 季度切换是 PREPARE、ACTIVATE、RELEASE-CHECK、发布并重启、EXPIRE、ARCHIVE/RECONCILE、PURGE。3 天宽限期内旧表可能仍有数据，但已退出正常在线路由；清空不是定时器无条件执行，必须确认待命表为空、归档成功、数量与校验和一致、无在途任务并人工批准。历史迁移则按旧主键游标幂等回填，按 tenant + quarter 对账，灰度切流并保留回滚窗。性能收益最终用同数据量、同索引和同负载下的 QPS、P95/P99、扫描行数、IO 与复制延迟证明。

## 5. STAR 结构

### S：Situation

- 核心表是点巡检记录。
- legacy 是单库单表，租户和历史数据持续累积。
- 高频访问是单租户、近半年时间范围。
- 普通索引优化不能解决单实例容量、归档边界和读写竞争的长期问题。

### T：Task

> 在尽量少改业务代码的前提下，降低单表和单实例压力，完整覆盖最近半年热查询，并让同一套产品代码支持轻量标准版与大客户隔离版，同时具备可迁移、可对账、可灰度和可回滚能力。

### A：Action

1. 用 `tenant_id` 作为固定租户分库键，用 `record_date` 作为季度分表键。
2. 设计 q1～q4 固定槽位和 current + previous 2 热窗口。
3. 用部署期开关在 standard/tenant 两份 YAML 间选择拓扑。
4. 配置一主一从、事务读主库与 Hint 强一致读。
5. 在 Service 层限制当前季度新增、过期/未来访问和缺分片键 SQL。
6. 设计游标回填、增量同步、按租户季度对账、灰度与回滚。

### R：Result

有真实证据时给出真实指标。只有 Demo 时说：

> Demo 验证了标准版所有租户固定 ds_0、大客户版偶/奇租户分库、日期精确季度表、跨 Q2/Q3 只命中两表、普通读/强一致读分别命中 replica/primary，以及过期、未来和历史新增保护。没有把本地吞吐当成生产收益。

## 6. 高频追问与参考答案

### Q1：为什么用 tenant_id 分库？

> 点巡检核心请求天然带当前租户，分库键和权限隔离边界一致，可把绝大多数请求裁剪到单库。代价是跨租户运营报表会跨库，通常应走预聚合、异步任务或数仓；头部租户也可能产生热点，所以生产要先看租户分布。

### Q2：为什么标准版不分库？

> 多数小客户只有 1～3 个租户，直接部署多个分片库会放大连接池、复制和运维成本。标准版仍通过 ShardingSphere 做季度分表与读写分离，只把 databaseStrategy 收敛为共享 ds_0；客户规模达到容量阈值时，在首次部署前选择大客户拓扑。

### Q3：`sharding.enabled=false` 是不是完全关闭 ShardingSphere？

> 不是。它只关闭 tenant_id 分库。应用仍使用 ShardingSphere-JDBC、`shardingsphere-standard.yaml`、四张固定季度表、Snowflake 和读写分离。变量名沿用简历口径，代码注释和启动日志会明确这个边界。

### Q4：为什么开关不能运行期切换？

> 标准版的奇数租户数据也在 ds_0；开启后同一个租户会被路由到 ds_1，但历史数据不会自动搬迁，查询会像“消失”。因此该参数只在应用启动时读取。有存量客户升级时要先建库表、回填、同步增量、对账并灰度切流。

### Q5：为什么 `sharding.database-sharding-key` 在 false 时仍然存在？

> 它明确声明的是数据库分片键，不是季度分表键。false 时所有租户固定走 ds_0，该字段不参与选库，但仍用于验证两种产品形态遵守同一个 tenant_id 契约；true 时它才参与数据库路由。季度分表键始终是 record_date。改成 asset_id 涉及 SQL、索引、算法和存量数据迁移，不能靠配置字符串完成。

### Q6：为什么 Demo 对 2 取模？

> 2 个库便于本地观察和手算，不是生产容量结论。生产库数要由增长、峰值负载、单库容量、租户倾斜、连接池乘法和运维成本决定。直接把 `%2` 改成 `%4` 会重映射历史租户，必须先迁移。

### Q7：为什么是固定四张季度表？

> 业务高频查询只要求最近半年，季度粒度能降低表数量和 DDL 运维复杂度；四个固定槽位又能让当前加前两季度使用三张表，留一张给下一季度准备。季度数据是否能维持目标 P99 必须用真实行宽、索引、并发和数据分布压测确认。

### Q8：三个季度是否等于固定时长？

> 不是。窗口是当前季度加前两个完整季度。季度初约覆盖 6 个多月，季度末接近 9 个月。它保证“最近半年完整可查”，不保证任意时刻恰好 9 个月。

### Q9：固定 q1～q4 如何避免跨年覆盖？

> 仅靠 Q1→q1 不够。`FixedQuarterShardingAlgorithm` 同时读取 `current-quarter`，只允许当前及前两个业务季度进入在线路由。数据库还有 `inspection_quarter_slot` 记录槽位绑定的 YYYYQn，清理和验收脚本会核对数据年份季度与绑定值。

### Q10：为什么不让表名直接带年份？

> 带年份能直观看懂历史，但会持续新增物理表和规则元数据。本方案把在线表数量固定为四张，将更早数据移入归档链路。代价是必须严格管理槽位状态、跨年校验和清理流程；如果业务必须长期在线查多年，则固定四槽位并不合适。

### Q11：季度切换如何做？

> 先确认下一槽位为空且 schema/index 一致，把该槽位元数据绑定为新季度并标记 ACTIVE；发布门禁再校验槽位绑定、表内数据年份季度和主从 GTID。门禁通过后发布新 `demo.sharding.current-quarter`、滚动重启并验证新季度路由，随后把最老季度标记 EXPIRED。宽限至少 3 天并完成归档、对账、流量栅栏与备份恢复检查后，最后人工批准清空，使它成为下一轮 EMPTY（空闲待命）槽位。

### Q12：3 天后会自动 TRUNCATE 吗？

> 不会。3 天只是最小宽限策略。清空前必须同时满足归档完成、数量/聚合/校验和一致、无在途读写、复制健康和人工确认；任何条件不满足都应阻断清理。

### Q13：宽限期是不是仍只有三张表有数据？

> 不一定。新季度已写入、最老季度仍待归档时，四张物理表可能都有数据，但只有 current + previous 2 是 ACTIVE 在线窗口，EXPIRED 表不再接受正常业务访问。

### Q14：为什么在线新增只允许当前季度？

> 防止普通请求把迟到数据写入即将归档清理的旧槽位。前两个季度可以在线查询，也允许受控迁移保留历史记录；业务若需要补录，应有独立审批、幂等、归档同步和审计流程。

### Q15：未来季度数据为什么不提前写待命表？

> 待命表尚未完成 ACTIVATE，提前写会破坏“空表可安全切换”的不变量。未来数据先拒绝或进入专门缓冲流程，季度发布完成后才允许正常写。

### Q16：分片算法具体是什么？

> 表路由是 `com.example.assetinspection.algorithm.quarter.FixedQuarterShardingAlgorithm`，根据 record_date 做精确和范围路由并校验热窗口；大客户分库是 `com.example.assetinspection.algorithm.tenant.TenantDatabaseShardingAlgorithm`，根据 tenant_id 选择 ds_0/ds_1。两者通过 ShardingSphere `CLASS_BASED` 标准算法接入。

### Q17：两种部署形态如何共用代码？

> Controller、Service、Mapper 始终访问逻辑表。启动时 `ProductDataSourceConfiguration` 根据开关选择 `shardingsphere-standard.yaml` 或 `shardingsphere-tenant.yaml`。差异只存在于数据源和规则配置，业务层不按物理库表写 if/else。

### Q18：ShardingSphere 的路由流程是什么？

> 它解析逻辑 SQL 和参数，先根据 tenant_id 选择目标库或共享库，再根据 record_date 裁剪季度表，改写逻辑表名；之后按 SQL、事务或 Hint 选择主从节点，执行物理 SQL并归并结果。

### Q19：如何证明精确路由而不是猜测？

> 先调用 `/api/debug/routes/expected` 手算，再看 sql-show 的 Logic SQL/Actual SQL，最后核对响应里的 `servedByServerId`、`servedByDatabase` 和物理验收脚本。预期、日志、响应和物理数据四项应一致。

### Q20：缺 record_date 会怎样？

> 有 tenant_id 时可以先确定一个逻辑库，但无法判断 q1～q4，教学反例会广播该库四张表。标准版始终是 ds_0；大客户版先按 tenant 选 ds_0 或 ds_1。`LIMIT 1` 也不能让路由阶段预知数据在哪张表。

### Q21：缺 tenant_id 会怎样？

> 正常 API 在进入 SQL 前拒绝缺租户上下文的请求。若底层 SQL 真缺 tenant_id，大客户模式可能跨库，标准版虽不跨库也失去行级租户隔离。分片路由不能替代认证授权。

### Q22：跨季度排序和聚合有什么成本？

> ShardingSphere 在每个命中表执行局部 SQL，再归并排序、聚合或分页。当前热窗口最多三张 ACTIVE 表，但范围越大，网络、内存和归并成本越高。大范围报表仍应走预聚合或分析系统。

### Q23：为什么游标分页优于深 OFFSET？

> `(record_date,id)` 复合游标和排序同序，能从上一页末尾继续，避免每个物理表扫描并丢弃大量前置行。代价是不能任意跳页，客户端要保存两个游标值。

### Q24：读写分离为什么会刚写后查不到？

> primary 提交后 replica 重放 binlog 有延迟，普通非事务读可能短暂看到旧值。事务内读由 YAML 配置为 PRIMARY，或用 Hint 临时强制主库；非关键历史查询可以接受最终一致。

### Q25：`@Transactional(readOnly=true)` 会自动走从库吗？

> 不会。`readOnly` 是 Spring 事务属性，不等于从库路由。本项目显式配置 `transactionalReadQueryStrategy: PRIMARY`，因此事务内查询走主库。

### Q26：Hint 有什么风险？

> Hint 基于线程上下文，必须用 try-with-resources/finally 清理，避免线程复用污染后续请求。它只改变读写节点，不能补齐 tenant_id 或 record_date，也不应成为所有读都走主库的捷径。

### Q27：ShardingSphere 会创建 MySQL 主从吗？

> 不会。Docker/MySQL 脚本创建复制账号、GTID 通道和只读副本；ShardingSphere 只配置已有 primary/replica 并路由。故障选主、RTO/RPO 和备份恢复是数据库基础设施职责。

### Q28：为什么使用 Snowflake？

> 多库多表不能依赖各表自增保证全局唯一。Snowflake 在应用中间件侧生成趋势递增 ID，减少中心序列瓶颈。生产必须保证 worker-id 唯一并处理时钟回拨；ID 本身不是本项目的路由键。

### Q29：分片键能直接 UPDATE 吗？

> 不应。修改 tenant_id 或 record_date 会改变物理位置，本质是跨节点搬迁。应受控地向新位置插入、校验，再删除旧位置并做补偿，或者创建纠正记录。

### Q30：历史数据怎么迁移？

> 旧单表按主键游标分批读取，目标写入保留旧 ID 并使用幂等策略；增量用 CDC 或 outbox/补偿双写。校验按 tenant + quarter 切片，至少比较 COUNT、状态聚合、数值汇总、分块校验和和抽样字段；先影子读，再灰度切流并保留回滚窗。

### Q31：为什么不能只比较总行数？

> 总数相同仍可能是一边少一批、另一边多一批。必须按租户和季度定位，再比较主键范围、聚合、分块 hash 与抽样数据，差异才能重放和修复。

### Q32：迁移为什么允许前两个季度，而普通 create 不允许？

> 迁移是受控历史回填，需要把仍处于 ACTIVE 热窗口的 Q1/Q2 数据补入目标表；普通 create 面向实时业务，只允许当前 Q3，避免历史补录污染即将退出的槽位。二者使用不同入口和校验语义。

### Q33：怎么扩到更多库？

> 不能直接改取模数。可以使用租户路由映射表让老租户保持位置、新租户进入新库，或分批搬迁指定租户。无论哪种都需要增量同步、校验、灰度和回滚。

### Q34：单表 500～1500 万行就一定稳定吗？

> 不一定。行数只是粗粒度指标；行宽、二级索引、SQL 选择性、热点、Buffer Pool、磁盘、并发和目标 P99 都会影响结果。正确说法是“把它作为容量档位，用客户典型数据和 Top SQL 压测达标后决定季度粒度”。

### Q35：分库分表后一定更快吗？

> 不一定。精确单分片查询通常受益于更小索引和负载分散；缺键广播、跨季度排序聚合可能更慢。必须在同等数据和负载下比较 QPS、P95/P99、扫描行、IO、路由扇出和复制延迟。
