# 架构与路由原理

本文解释当前 Demo 的真实实现：先保留一套改造前的单库单表作为对照，再用同一套业务代码支持两种产品部署形态。核心链路是：**部署时选择拓扑 → 按租户选库（可选）→ 按业务日期选固定季度表 → 按 SQL 类型与事务状态选主从 → 改写、执行与归并**。

## 1. `legacy` 到底是什么意思

`legacy` 是英文“遗留的、原有的”。在这个项目中，它专指**引入 ShardingSphere-JDBC 之前的旧架构基线**：

- 一个 MySQL 库 `asset_legacy`；
- 一张物理表 `inspection_record`；
- 应用通过普通 MySQL JDBC 直接连接；
- 没有租户分库、季度分表和读写分离。

它不是第三种产品交付方案，也不是“低配版”。保留 `legacy` 的目的有三个：

1. 演示改造前后代码和路由行为的差异；
2. 作为存量数据迁移的只读源端；
3. 用同样的数据和 SQL 做性能基线对比。

启动旧基线时显式使用：

```bash
mvn spring-boot:run -Dspring-boot.run.profiles=legacy
```

真正交付给客户时使用 `product`，并在部署配置里决定是否开启租户分库。

## 2. 三种运行视角，但只有两种产品形态

| 运行视角 | 配置 | 数据拓扑 | ShardingSphere-JDBC | 用途 |
|---|---|---|---|---|
| 改造前对照 | `legacy` profile | 单库单表 | 不经过 | 学习、迁移源端、基线压测 |
| 标准版 | `product` + `sharding.enabled=false` | 一个共享库、四张固定季度表、一主一从 | 经过 | 默认轻量交付 |
| 大客户版 | `product` + `sharding.enabled=true` | 按 `tenant_id` 分库、每库四张固定季度表、每库一主一从 | 经过 | 需要租户物理隔离或更高容量的客户 |

最容易说错的一点是：

> `sharding.enabled=false` 只表示“关闭 tenant_id 分库”，并不表示关闭 ShardingSphere-JDBC。

标准版仍由 ShardingSphere-JDBC 完成季度分表、Snowflake 主键、DML 审计和读写分离。这样 Controller、Service、Mapper 只维护一套，变化被限制在部署配置和路由规则中。

## 3. 为什么开关只能在部署阶段决定

正式入口使用：

```properties
sharding.enabled=${SHARDING_ENABLED:false}
sharding.database-sharding-key=${DATABASE_SHARDING_KEY:tenant_id}
```

`ProductDataSourceConfiguration` 在 Spring 启动时读取一次配置：

- `false` 加载 `shardingsphere-standard.yaml`；
- `true` 加载 `shardingsphere-tenant.yaml`。

随后连接池、实际数据节点和分片算法一起初始化。应用运行中不会替换 DataSource，也不会监听配置热更新。

如果一个已运行的标准版直接把 `false` 改为 `true`，历史数据仍在共享库，新的路由规则却会去租户分片库查找，结果就是历史数据“消失”。因此升级必须经过建库建表、存量回填、增量追平、对账、切流和回滚演练，然后通过发布重启启用新规则。

`sharding.database-sharding-key` 明确表示数据库分片键，不是季度分表键。当前代码启动时只接受 `tenant_id`；`false` 时它不参与选库、只承担契约自检，`true` 时才用于租户数据库路由。季度分表键固定为 `record_date`。更换任何分片键都属于数据架构迁移。

## 4. 数据模型与分片键

点巡检记录的高频访问模式是：

- 某租户查询当前及最近两个季度的巡检明细；
- 某租户按时间范围统计正常、异常、已修复数量；
- 使用记录 ID、租户和业务日期查询或更新单条记录；
- 少量平台运营任务做跨租户汇总，但不属于在线核心链路。

因此选择：

- `tenant_id`：大客户版的分库键，也是所有形态中的数据权限条件；
- `record_date`：固定季度表的分表键；
- `id`：由 Snowflake 生成全局主键，但 ID 本身不编码库表位置。

核心字段和索引如下：

| 字段 | 作用 | 约束 |
|---|---|---|
| `id` | 全局记录 ID | 新拓扑由 Snowflake 生成；迁移时保留旧 ID |
| `tenant_id` | 分库键与租户过滤 | 非空；在线 SQL 必须携带 |
| `record_date` | 季度分表键 | 与 `inspected_at` 的日期部分一致 |
| `request_id` | 请求幂等号 | 当前唯一约束只在一个物理季度槽位内有效 |
| `status` | 巡检状态 | Java 枚举和数据库 CHECK 双重约束 |
| `version` | 乐观锁版本 | 更新条件包含期望版本 |

每张季度表的主要索引是：

```text
PRIMARY KEY (id)
UNIQUE (tenant_id, request_id)
INDEX (tenant_id, record_date, id)
INDEX (tenant_id, status, record_date)
INDEX (tenant_id, asset_id, record_date)
```

索引是否适合真实客户仍要用 Top SQL、数据分布和 `EXPLAIN ANALYZE` 验证。一个物理表内的唯一键不能自动保证跨季度全局幂等；若业务要求 `request_id` 永久唯一，需要独立幂等表或其他一致性组件。

## 5. 逻辑表与物理表

业务代码和 MyBatis 始终访问逻辑表：

```text
inspection_record
```

四个固定物理槽位是：

```text
inspection_record_q1
inspection_record_q2
inspection_record_q3
inspection_record_q4
```

映射永久固定：

```text
自然季度 Q1 → q1
自然季度 Q2 → q2
自然季度 Q3 → q3
自然季度 Q4 → q4
```

表名不包含年份。因此 `2026Q1` 与 `2027Q1` 都会计算出 `q1`，但它们绝不能同时存在于该槽位。安全复用依靠三季度热窗口、槽位元数据和受控清理流程，而不是只靠取模公式。

两种产品拓扑分别声明：

```text
标准版：ds_0.inspection_record_q1 ... ds_0.inspection_record_q4
大客户版：ds_0 / ds_1 各有 inspection_record_q1 ... inspection_record_q4
```

`ds_0`、`ds_1` 是 ShardingSphere 的逻辑读写组：

```text
ds_0 = ds0_primary + ds0_replica
ds_1 = ds1_primary + ds1_replica
```

## 6. 产品启动时如何选择 YAML

`ProductDataSourceConfiguration` 创建唯一的逻辑 DataSource，并根据部署开关选择 JDBC URL：

```text
false → jdbc:shardingsphere:classpath:shardingsphere-standard.yaml
true  → jdbc:shardingsphere:classpath:shardingsphere-tenant.yaml
```

两份 YAML 的共同部分是：

- 四个固定季度实际节点；
- `FixedQuarterShardingAlgorithm`；
- Snowflake 主键；
- `DML_SHARDING_CONDITIONS` 审计器；
- 一主一从读写组和 `transactionalReadQueryStrategy: PRIMARY`。

区别只有数据库拓扑：标准版不配置数据库分片策略；大客户版增加 `TenantDatabaseShardingAlgorithm`。

这就是“一套代码适配两种交付形态”的边界：业务层相同，数据源规则在启动时固定。

## 7. 大客户版：按 tenant_id 选库

`shardingsphere-tenant.yaml` 通过 `CLASS_BASED` 加载：

```text
com.example.assetinspection.algorithm.tenant.TenantDatabaseShardingAlgorithm
```

Demo 配置 `shard-count=2`，算法使用 `floorMod(tenantId, 2)`：

```text
偶数 tenant_id → ds_0
奇数 tenant_id → ds_1
```

算法对等值条件能精确选择一个库；租户范围条件无法安全裁剪取模分片，只能返回全部候选库。因此在线 SQL 应使用等值租户条件。

标准版没有 `databaseStrategy`，所有租户都进入 `ds_0`，但 SQL 中仍必须包含：

```sql
WHERE tenant_id = #{tenantId}
```

这是因为“物理上共享一个库”和“业务上允许跨租户访问”是两回事。`TenantContextHolder` 只保存当前租户，Mapper 的 SQL 仍要显式过滤；认证层还必须验证登录用户确实属于该租户。

直接把 `shard-count` 从 2 改成 4 会让大量租户重新映射，不能作为普通扩容操作。生产可使用租户路由映射表、选定租户迁移或版本化迁移态规则，但都需要数据搬迁与对账。

## 8. 固定季度算法如何路由

两份 YAML 都通过 `CLASS_BASED` 加载：

```text
com.example.assetinspection.algorithm.quarter.FixedQuarterShardingAlgorithm
```

算法初始化需要：

```properties
demo.sharding.current-quarter=${CURRENT_QUARTER:2026Q3}
demo.sharding.retained-quarter-count=3
```

若当前季度是 `2026Q3`，在线窗口为：

```text
[2026-01-01, 2026-10-01)
```

也就是 `2026Q1、2026Q2、2026Q3`，对应 `q1、q2、q3`；`q4` 处于待命状态。

需要准确表达“九个月”：三个完整自然季度在物理表定义上覆盖九个日历月，但站在季度中的某一天回看，实际可查询时长约为 6～9 个月，不是每天都固定保存最近 270 天。

### 8.1 精确日期

`record_date=2026-05-12` 属于 `2026Q2`，算法先确认日期位于热窗口，再返回：

```text
inspection_record_q2
```

这一步先做年份和窗口校验非常关键。否则一条 `2025Q2` 的迟到数据也可能因为“同为 Q2”误写进正在承载 `2026Q2` 的 q2。

### 8.2 时间范围

查询采用半开区间：

```sql
record_date >= '2026-03-20'
AND record_date < '2026-07-02'
```

实际覆盖 `2026Q1、2026Q2、2026Q3`，因此只访问 `q1、q2、q3`。若起点早于热窗口，或终点超过当前季度结束边界，算法直接拒绝，而不是静默截断结果。

### 8.3 普通新增只允许当前季度

路由算法允许精确访问三个热季度，便于历史详情、更新、迁移和对账；业务服务对普通创建再增加一层约束：

```text
普通新增 record_date 必须属于 current-quarter
```

若当前是 `2026Q3`，正常创建只能写 q3。前两个季度的迟到补录必须走受控补录或迁移流程，不能通过普通 API 把数据写进即将退出热窗口的槽位。

## 9. CURRENT_QUARTER 为什么不能读系统时钟

当前季度由部署参数明确给出，而不是每个 JVM 在午夜自行判断。原因是：

- 多实例时钟或发布时间可能不一致；
- 物理表是否已清空、元数据是否已绑定是数据库运维事实；
- 规则切换需要与归档、校验、发布和回滚一起执行。

Spring 先解析 `demo.sharding.current-quarter`，再把同一个值写入 JVM 属性供 ShardingSphere YAML 使用，避免业务校验认为当前是 Q4，而算法仍按 Q3 路由。

季度推进需要修改部署环境的 `CURRENT_QUARTER` 并滚动重启。发布期间新旧实例对可写季度的认识不同，所以应先准备目标槽位，再通过摘流、分批重启、健康检查和快速收敛控制混跑时间，不能把它理解成无协调的运行期热切换。

## 10. 四张表的生命周期边界

数据库中有 `inspection_quarter_slot` 元数据表，显式记录：

- `physical_table`：q1～q4；
- `bound_quarter`：当前绑定的 `YYYYQn`；
- `slot_status`：`EMPTY / ACTIVE / EXPIRED`；
- `activated_at / expired_at / cleanup_after`：启用、过期和最早清理时间。

常态下三个槽位为 `ACTIVE`，一个槽位为 `EMPTY`。季度切换时会短暂出现：新季度槽位已经 `ACTIVE`，最老季度槽位为 `EXPIRED`，四张表都有数据。所谓“一张表闲置待命”描述的是清理完成后的稳定状态，不是三天宽限期内的每一秒。

ShardingSphere 只负责路由，不负责归档、修改槽位元数据或 `TRUNCATE`。三天宽限是最早允许清理的业务保护线，不是定时自动删除命令；真正清理前仍要检查归档、对账、在途任务、备份和主从复制。完整流程见《迁移灰度与生产化》。

## 11. 读写分离

每个逻辑分片库配置一个写库和一个读库：

```text
INSERT / UPDATE / DELETE → primary
非事务普通 SELECT → replica
事务内 SELECT → primary
HintManager.setWriteRouteOnly() → primary
```

`transactionalReadQueryStrategy: PRIMARY` 保证同一事务内写后读能够看到刚写入的数据。普通历史列表可以接受从库最终一致；刚写后回显、关键状态确认应使用事务读或 Hint 强制主库。

ShardingSphere-JDBC 管理的是应用侧“把哪类 SQL 发往哪个节点”，不会替 MySQL 创建复制关系，也不会独立完成故障选主。复制、备份、恢复、延迟监控和高可用切换仍由数据库基础设施负责。

## 12. 一条 SQL 的完整路由示例

大客户版中，tenant 3 查询：

```sql
SELECT status, COUNT(*)
FROM inspection_record
WHERE tenant_id = 3
  AND record_date >= '2026-04-01'
  AND record_date < '2026-08-01'
GROUP BY status;
```

执行过程是：

1. SQL 解析器提取 `tenant_id=3`，自定义租户算法选择 `ds_1`；
2. 固定季度算法判断范围覆盖 Q2 和 Q3，选择 q2、q3；
3. 普通非事务查询从 `ds1_replica` 读取；
4. 逻辑表名改写成两条物理 SQL；
5. 各表执行局部聚合，ShardingSphere 再归并为最终结果。

标准版执行相同 SQL 时跳过第 1 步的分库计算，固定进入 `ds_0`，其余过程一致。

## 13. 缺少分片条件会发生什么

在标准版：

- 有 `record_date`：可裁剪到一个或少数季度表；
- 缺 `record_date`：可能扫描共享库的四张表。

在大客户版：

- 有 `tenant_id`、缺 `record_date`：选定一个库后扫描四张表；
- 缺 `tenant_id`、有 `record_date`：两个库都访问相应季度表；
- 两者都缺：最多访问两个库的全部四张表。

因此在线 SQL 必须携带租户和完整日期条件。DML 审计器会阻止部分缺分片条件的误更新或误删除；SELECT 广播仍需通过接口设计、代码评审和路由扇出监控约束。

## 14. Snowflake 主键与写后回读

应用正常 INSERT 不传 `id`，ShardingSphere 在 SQL 改写阶段生成 Snowflake ID。这个项目随后在同一事务中，按 `tenant_id + record_date + request_id` 从主库精确回读新记录。

这样做是因为当前依赖组合下，MyBatis `useGeneratedKeys` 与 ShardingSphere 逻辑生成键的兼容性实验会出现多键异常。它是这个 Demo 的明确兼容方案，不是所有版本都必须采用的固定写法。

Snowflake 的边界包括：

- 多应用实例必须分配不同 worker-id；
- 需要监控时钟回拨；
- 只凭 ID 不能推导租户库和季度表；
- 对外暴露趋势 ID 是否可接受需做安全评估。

## 15. “单表 500～1500 万稳定”应如何表述

可以把 500～1500 万行作为客户画像和容量设计区间，但不能承诺“达到这个行数就一定稳定”。查询性能还取决于：

- 行宽、索引大小和 Buffer Pool 命中率；
- 租户和状态值的数据倾斜；
- 查询是否带租户与日期，是否回表、排序或聚合；
- 并发、磁盘、CPU、连接池和复制延迟；
- P95/P99 目标，而不只是平均耗时。

正确说法是：

> 多数标准版客户只有 1～3 个租户，预计单季度表约 500～1500 万行；我们以此作为容量分界的初始假设，并用等量数据、真实索引和 Top SQL 压测验证 P95/P99。超过阈值或出现头部租户热点时，再选择租户分库形态。

## 16. 如何证明路由真的正确

一次实验至少同时看三类证据：

1. `/api/debug/routes/expected` 给出的预期库和季度槽位；
2. ShardingSphere 的 `Logic SQL / Actual SQL` 日志；
3. 接口返回的 `servedByServerId / servedByDatabase` 和物理表查询结果。

三者一致才能证明“库、表、主从”都按规则工作。接口返回 200 只能证明请求成功，不能证明没有广播。

## 17. 面试概括

> 我们保留 legacy 单库单表作为迁移源和压测基线；产品运行统一走 ShardingSphere-JDBC。标准版部署时 `sharding.enabled=false`，仍使用单库、四张固定季度表和读写分离；大客户部署时设为 true，由自定义算法按 tenant_id 分库。record_date 通过 FixedQuarterShardingAlgorithm 固定映射到 q1～q4，只保留 current-quarter 和前两个季度，普通新增只允许当前季度。开关和 CURRENT_QUARTER 都是启动参数，变更必须配合数据迁移或季度轮转后重启，不支持运行期热切换。四张表的归档、三天宽限和清理由受控运维流程负责，ShardingSphere 只做规则路由。
